## 二、阶段 0：可行性验证

> **先于任何一行功能代码**

### 2.1 验证目标

确认 WebView2 WebGL 性能能否满足 3D 主题的 60fps 底线。

### 2.2 验证内容

| 测试项 | 方法 | 通过标准 |
|--------|------|---------|
| 压力场景 FPS | 10 万三角面 + 多贴图 + Bloom 后处理 | 稳定 ≥ 60 FPS |
| Context 创建耗时 | 计时 `getContext("webgl2")` + 首帧渲染 | < 500ms |
| Context Lost 恢复 | 调用 `loseContext()` → 重建 → 渲染 | < 3s 恢复 |
| 内存基线 | JS Heap 空闲 / 加载场景后 / 运行 5min 后 | 稳定不持续增长 |
| 虚拟适配器 | 远程桌面 / VM 环境下的行为 | 优雅降级非崩溃 |

### 2.3 失败处理

如果验证不通过（< 60 FPS 或频繁崩溃）：
- 评估是否为测试代码问题 → 复测
- 确认是 WebView2 硬限制 → 评估降帧运行（30 FPS）是否可接受
- 完全无法满足 → 重新评估技术路线

## 三、阶段 1：渲染内核开发

### 3.1 开发内容

| 模块 | 产出 |
|------|------|
| RenderManager | 全局唯一 WebGL context 管理 |
| SceneManager | 场景加载/卸载/切换 |
| ResourceCache | GPU 资源生命周期管理 |
| ShaderCompiler | 着色器编译 + 缓存 + 错误处理 |
| MetricsCollector | 性能指标采集 |
| CircuitBreaker | 熔断器状态机 |

### 3.2 隔离测试

每个模块写单元测试（不依赖画布），验证：
- 创建 → 正常
- 销毁 → 无泄漏
- 异常输入 → 不崩溃

### 3.3 首个测试场景

使用极简硬编码场景（一个立方体 + 基础 PBR 着色器），不依赖任何主题素材，验证渲染管线跑通。

## 四、阶段 2：素材生产 + 前端集成（并行）

### 4.1 素材生产线

```
线 A：Cyber-Girl-Apartment

  ├─ 管线脚本处理
  └─ 素材校验 → 修正 → 冻结

线 B：古风水墨（与 A 同步，同节奏）
  ├─ AI 生成提示词 → 3D 建模工具 → glTF 导出
  ├─ 贴图生成 → KTX2 转换
  ├─ 管线脚本处理
  └─ 素材校验 → 修正 → 冻结
```

### 4.2 前端集成

| 组件 | 产出 |
|------|------|
| ThreeDCanvas | 画布组件 |
| ThreeDErrorBoundary | 异常边界 |
| LoadingOverlay | 加载进度 |

### 4.3 Lockstep 检查点

**执行策略**：Cyber-Girl-Apartment 做完整实现，古风水墨仅做骨架验证（极简场景 + 关键 shader 参数 + 1-2 个交互）。骨架验证的目标是确认底层接口的通用性，不要求完整素材产出。

**每周一次**双主题交叉验证：

```
本周 Cyber-Girl 实现了什么 → 古风水墨用同样的底层接口能实现吗？
    ├─ 能 → 底层 OK
    └─ 不能 → 底层接口调整 → 本周修正
```

| 检查点 | 验证内容 |
|--------|---------|
| CK1：渲染管线 | 两种风格的场景能正确加载 + 渲染（哪怕素材不完整） |
| CK2：着色器参数 | Cyber 发光参数集 + 水墨墨染参数集能通过同一 UBO 接口注入 |
| CK3：交互系统 | 点击全息面板 vs 点击卷轴的交互流程走同一 EventBus |
| CK4：动画状态机 | Cyber 角色的 idle/greet vs 水墨角色的 idle/bow 走同一动画控制器 |
| CK5：粒子系统 | 数字雨 vs 水墨飞溅走同一粒子管理器 |
| CK6：打包流程 | 两个主题能分别打出合法 NV3D 文件 + 客户端正确加载 |

## 十、双机协同约定

### 10.1 问题

单人 + 办公室/家里双机开发，代码和规范通过 Git 同步，但**进度状态不在 Git 里**——办公室写到一半的代码，回家后不知道做到哪了。

### 10.2 方案

进度文件 `docs/webgl3d-spec/PROGRESS.md` 随代码一起纳入 Git。文件结构见 [PROGRESS.md](PROGRESS.md)。

### 10.3 工作流

```
【机器 A — 工作结束前】
  1. 打开 PROGRESS.md
  2. 当前在做的条目：补充"写到哪了 + 下一步"
  3. 已完成的条目：从 [~] 改为 [x]，移到"已完成"
  4. npm run push "简述今日进度"

【机器 B — 工作开始前】
  1. npm run pull
  2. 打开 PROGRESS.md → 看"当前在做"
  3. 打开 00_索引末尾的"文档变更日志" → 看有无规范变更
     若有变更 → 检查 PROGRESS.md 中"已完成"和"偏离记录"是否受影响
  4. 继续写代码
```

### 10.4 规范变更与进度的联动

```
规范变了（如 05_渲染管线 新增了恢复时间预算）
    ↓
1. 修改规范文档
2. 在 00_索引末尾"文档变更日志"记一行
3. 打开 PROGRESS.md
4. 看"已完成"和"偏离记录"中有没有引用该规范的条目
    ├─ 有 → 在"当前在做"追加一条：[~] 回改 XXX — 规范 §X.Y 已变更
    └─ 无 → 不操作
5. npm run push
```

另一台机器 pull 后看到 Changelog 有新增 → 执行同样的检查。

### 10.5 纪律

- **必须**：切换机器前更新 PROGRESS.md 并 push
- **必须**：规范文档有实质性变更时，在 Changelog 记一行
- **建议**：每完成一个模块就更新一次（不要攒到切换机器前才想起来）
- **不必**：每次 commit 都改 PROGRESS.md——功能写一半没完成，commit 可以不带进度更新
