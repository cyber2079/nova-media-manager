## 二、阶段 0：可行性验证

> **先于任何一行功能代码**

### 2.1 验证目标

确认 WebView2 WebGL 性能能否满足 3D 主题的 60fps 底线。

### 2.2 验证内容

| 测试项 | 方法 | 通过标准 |
|--------|------|---------|
| 压力场景 FPS | 10 万三角面 + 多贴图 + Bloom 后处理 | 稳定 ≥ 60 FPS |
| Context 创建耗时 | 计时 `getContext("webgl2")` + 首帧渲染 | < 500ms |
| Context Lost 恢复 | 调用 `loseContext()` → 重建 → 渲染 | < 3s 恢复 |
| 内存基线 | JS Heap 空闲 / 加载场景后 / 运行 5min 后 | 稳定不持续增长 |
| 虚拟适配器 | 远程桌面 / VM 环境下的行为 | 优雅降级非崩溃 |

### 2.3 失败处理

如果验证不通过（< 60 FPS 或频繁崩溃）：
- 评估是否为测试代码问题 → 复测
- 确认是 WebView2 硬限制 → 评估降帧运行（30 FPS）是否可接受
- 完全无法满足 → 重新评估技术路线

### 2.4 产出

- `docs/webgl3d-spec/feasibility-report.md` — 性能基准数据
- 决定 05/19 文档中的硬性阈值

## 三、阶段 1：渲染内核开发

### 3.1 开发内容

| 模块 | 产出 |
|------|------|
| RenderManager | 全局唯一 WebGL context 管理 |
| SceneManager | 场景加载/卸载/切换 |
| ResourceCache | GPU 资源生命周期管理 |
| ShaderCompiler | 着色器编译 + 缓存 + 错误处理 |
| MetricsCollector | 性能指标采集 |
| CircuitBreaker | 熔断器状态机 |

### 3.2 隔离测试

每个模块写单元测试（不依赖画布），验证：
- 创建 → 正常
- 销毁 → 无泄漏
- 异常输入 → 不崩溃

### 3.3 首个测试场景

使用极简硬编码场景（一个立方体 + 基础 PBR 着色器），不依赖任何主题素材，验证渲染管线跑通。

## 四、阶段 2：素材生产 + 前端集成（并行）

### 4.1 素材生产线

```
线 A：Cyber-Girl-Apartment
  ├─ AI 生成提示词 → 3D 建模工具 → glTF 导出
  ├─ 贴图生成 → KTX2 转换
  ├─ 管线脚本处理
  └─ 素材校验 → 修正 → 冻结

线 B：古风水墨（与 A 同步，同节奏）
  ├─ AI 生成提示词 → 3D 建模工具 → glTF 导出
  ├─ 贴图生成 → KTX2 转换
  ├─ 管线脚本处理
  └─ 素材校验 → 修正 → 冻结
```

### 4.2 前端集成

| 组件 | 产出 |
|------|------|
| ThreeDCanvas | 画布组件 |
| ThreeDErrorBoundary | 异常边界 |
| LoadingOverlay | 加载进度 |
| DialogBox | 剧情弹窗 |
| InteractionHint | 交互提示 |
| ThemeSwitcherPanel | 主题切换面板 |
| PermissionGate | 权限拦截 |
| threeDStore | 状态管理 |

### 4.3 Lockstep 检查点

**执行策略**：Cyber-Girl-Apartment 做完整实现，古风水墨仅做骨架验证（极简场景 + 关键 shader 参数 + 1-2 个交互）。骨架验证的目标是确认底层接口的通用性，不要求完整素材产出。

**每周一次**双主题交叉验证：

```
本周 Cyber-Girl 实现了什么 → 古风水墨用同样的底层接口能实现吗？
    ├─ 能 → 底层 OK
    └─ 不能 → 底层接口调整 → 本周修正
```

| 检查点 | 验证内容 |
|--------|---------|
| CK1：渲染管线 | 两种风格的场景能正确加载 + 渲染（哪怕素材不完整） |
| CK2：着色器参数 | Cyber 发光参数集 + 水墨墨染参数集能通过同一 UBO 接口注入 |
| CK3：交互系统 | 点击全息面板 vs 点击卷轴的交互流程走同一 EventBus |
| CK4：动画状态机 | Cyber 角色的 idle/greet vs 水墨角色的 idle/bow 走同一动画控制器 |
| CK5：粒子系统 | 数字雨 vs 水墨飞溅走同一粒子管理器 |
| CK6：打包流程 | 两个主题能分别打出合法 NV3D 文件 + 客户端正确加载 |

| bridge/ 封装层 | 前端 tauriCommands.ts |

### 5.2 测试

```
- 打开 v1 普通主题 → 正常（兼容）
- 打开 v2 3D 主题 → Manifest 解析正确
- 读取特定 Block → hash 匹配
- 打开已损坏（hash 不匹配）的包 → 优雅降级
- 签名无效的包 → 拒绝加载
- 存档读写 → 数据正确 + 未污染现有表
```

## 六、阶段 4：NV3D 打包 + 联调

### 6.1 打包脚本

`scripts/webgl3d/pack_nv3d.py` 完成开发，对两个素材目录分别打包。

### 6.2 联调测试

```
完整流程测试：
1. 运行 pack_nv3d.py → 生成 .nv3d 文件
2. 上传至本地测试服务器
3. 客户端打开设置 → 导入主题
4. 激活主题 → 完整加载 → 可交互
5. 切换主题 → 旧主题卸载 + 新主题加载
6. 关闭 3D 模块 → 壁纸恢复
7. 重新打开 → 存档保留
```

## 七、阶段 5：测试 + 修复

### 7.1 测试矩阵

| 测试类别 | 测试场景 | 通过标准 |
|---------|---------|---------|
| **隔离测试** | Feature Flag = false → 启动 | 3D 代码 0 字节加载 |
| **隔离测试** | 3D 模块有语法错误 → 启动 | 主应用正常 |
| **功能测试** | 完整主题加载流程 | 低模 < 5s 可交互 |
| **功能测试** | 主题热切换 | < 10s 完成 |
| **异常测试** | 加载中 kill WebGL context | Error Boundary 捕获 + 降级 |
| **异常测试** | Shader 编译失败 | Fallback 着色器 + 场景继续渲染 |
| **异常测试** | Worker 线程崩溃 | 主线程继续 + 降级 |
| **异常测试** | NV3D 文件损坏 | 优雅拒绝 + 提示 |
| **异常测试** | OOM 触发 | 熔断 + 模块禁用 + 壁纸恢复 |
| **场景测试** | 睡眠唤醒 | Context 恢复或降级 |
| **场景测试** | 窗口 resize / DPI 变化 | Canvas 正确适配 |
| **场景测试** | 远程桌面连接 | 优雅降级 |
| **场景测试** | 会员过期 | 模块禁用 + 数据保留 |
| **场景测试** | 续费恢复 | 模块启用 + 存档恢复 |
| **场景测试** | Free 用户 | 不显示 3D 入口 |

### 7.2 Bug 修复流程

- 每个 Bug 修复后跑完整回归测试（至少隔离测试 + 功能测试）
- 修复不引入新的 lint/typecheck 错误
- `npm run typecheck` 全量通过后再合入

## 八、阶段 6：交付 + 服务端部署

### 8.1 客户端交付

- 代码合入 master 分支
- 3D 模块代码通过 `npm run tauri:build` 正常构建
- Feature Flag 合入时确保为 `false`
- 发布 MSI 安装包

### 8.2 服务端交付

- 上传 NV3D 主题文件至 CDN
- 更新服务端元数据 API（新增 3D 主题列表）
- 配置 `webgl3d_settings` 数据库表（服务端如存储）

### 8.3 发布后监控

- 关注用户反馈中的崩溃日志（见 02 文档第六节）
- 首次发布仅对部分用户开放（灰度），稳定后逐步扩大

## 九、Lockstep 冻结标准

在阶段 5 测试完成后、阶段 6 交付前，确认：

- [ ] Cyber-Girl-Apartment 全部功能通过测试
- [ ] 古风水墨全部功能通过测试
- [ ] 两个主题**未对底层通用代码做任何专属修改**
- [ ] 底层接口在两个主题的 manifest 中均能完整描述
- [ ] 可以新增第三个虚构主题（如"星际飞船"），manifest 编写不需要改动底层代码

以上全部通过 → 底层通用规范冻结。
