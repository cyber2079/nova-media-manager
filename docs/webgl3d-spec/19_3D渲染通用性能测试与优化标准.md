## 一、硬件资源阈值

### 1.1 硬性红线

| 指标 | 红线值 | 测量方式 | 超限行为 |
|------|--------|---------|---------|
| 单帧渲染时间 | ≤ 33ms（30 FPS 底线） | `performance.now()` delta | 触发降级评估 |
| JS Heap | ≤ 512 MB | `performance.memory.usedJSHeapSize` | 触发 OOM 预警 |
| 三角面数 | ≤ 100,000 | 场景加载时统计 | 降画质 |
| DrawCall | ≤ 200 | 每帧统计 | 降画质 |
| 纹理总尺寸 | ≤ 512 MB 等效显存 | 按分辨率 × 格式估算 | 降纹理分辨率 |
| WebGL context 创建 | ≤ 1s | 计时 | 重试或降级 |

### 1.2 目标值（推荐）

| 指标 | 目标值 | 说明 |
|------|--------|------|
| FPS | 60（桌面端） | 稳态 |
| 帧时间 p95 | ≤ 20ms | 留 16.6ms 预算的余量 |
| 内存增长率 | 5min 内增长 < 50MB | 无持续泄漏 |
| 场景加载（到可交互） | ≤ 5s | Phase 2 完成 |
| 主题热切换 | ≤ 10s | 含资源释放+加载 |

## 三、资源泄漏检测

### 3.1 检测方法

| 资源类型 | 检测工具/方法 | 检测频率 |
|---------|-------------|---------|
| WebGL Texture | 遍历 ResourceCache，统计 `gl.isTexture()` 仍有效的对象数 | 场景切换后 |
| WebGL Buffer | 同上，统计 `gl.isBuffer()` 有效对象数 | 场景切换后 |
| WebGL Shader/Program | 同上，统计 `gl.isProgram()` / `gl.isShader()` | 场景切换后 |
| WebGL Framebuffer/Renderbuffer | 同上 | 场景切换后 |
| JS Event Listener | Chrome DevTools Memory 面板 → Event Listeners | 手动 |
| Web Worker | `worker.terminate()` 调用计数对比 | 主题切换后 |
| DOM Node | Canvas 卸载后检查残留 DOM 引用 | 模块卸载后 |

### 3.2 泄漏判定

```
场景切换后检查：
    切换前资源数 = ResourceCache.count()
    执行热切换
    切换后资源数 = ResourceCache.count()
    
    如果 切换后资源数 > 旧主题资源数 + 新主题资源数（允许 10% 偏差）
        → 判定为资源泄漏
        → 自动触发全量 dispose + gl.flush()
        → 记录警告日志
```

### 3.3 周期性巡查

- 每 30s 自动巡查一次 ResourceCache
- 发现有超过 30s 未被引用的可释放资源 → 自动 dispose
- 仅对短期缓存中的资源执行，不触碰当前场景的活跃资源

## 四、渲染开销优化指南

### 4.1 优化优先级

| 优先级 | 优化项 | 预期收益 |
|--------|--------|---------|
| 1 | 视锥剔除（Frustum Culling） | 减少 30-70% DrawCall |
| 2 | LOD 切换（远近不同精度模型） | 减少 20-50% 三角面 |
| 3 | 纹理 Mipmap | 减少远处纹理带宽 + 提高质量 |
| 4 | 合批渲染（Instanced Rendering） | 减少 DrawCall（粒子/重复物件） |
| 5 | 阴影贴图分辨率动态调整 | 减少阴影 pass 开销 |
| 6 | 后处理降分辨率（如 1/2 分辨率渲染后 upscale） | 减少片元着色器负载 |

### 4.2 优化不容牺牲的底线

| 不能牺牲 | 替代方案 |
|---------|---------|
| 禁用视锥剔除来"简化代码" | — |
| 取消 mipmap ← 节省显存 | 降低贴图分辨率 |
| 降到 30fps 以下 ← 省电 | 降低画质参数 |

### 5.1 最低配置

| 组件 | 最低要求 | 说明 |

| GPU | DirectX 11 兼容 / OpenGL ES 3.0 | WebGL 2.0 要求 |
| 显存 | 512 MB | 共享内存可接受 |
| 内存 | 4 GB | — |
| 屏幕分辨率 | 1366×768 | — |

### 5.2 低配自适应

检测到低配设备（通过 WebGL renderer 字符串 + 初始 FPS 判断）：

```
检测到低配设备
    ├─ 初始画质设为 "low"
    ├─ 默认 30 FPS
    ├─ 禁用 Bloom / SSAO 等重后处理
    ├─ 仅加载低精度模型（跳过 HD）
    └─ 用户可手动提升画质（接受风险提示）
```

### 5.3 虚拟适配器

检测到虚拟显示适配器时（Microsoft Basic Render Driver / GDI Generic）：
- 直接走静态降级，不尝试 3D 渲染
- 提示用户"当前环境不支持 3D 渲染"

## 六、功耗测试

### 6.1 测试方法

| 测试项 | 方法 | 通过标准 |
|--------|------|---------|
| 空闲功耗 | 场景加载完毕，无交互，5min | GPU 使用率 < 10% |
| 活跃功耗 | 持续旋转场景 + 粒子动画，5min | GPU 频率不持续 > 80% |
| 失焦功耗 | 窗口最小化，3min | requestAnimationFrame 停止 |
| 恢复功耗 | 失焦 → 恢复焦点 | 1s 内恢复正常帧率 |

### 6.2 测试环境

- 笔记本（非台式机）— 功耗对电池续航有直接影响
- 使用 GPU-Z / HWiNFO 监控 GPU 频率和功耗

## 八、模块满载 + 崩溃场景专项测试

### 8.1 满载测试

| 测试 | 操作 | 通过标准 |
|------|------|---------|
| 长时间运行 | 连续运行 2h，期间交互操作 | FPS 不衰减，内存增长 < 100MB |
| 快速操作 | 1min 内执行 50 次交互 | 无卡顿，无 quene 堆积 |

### 8.2 崩溃场景

| 测试 | 触发方式 | 通过标准 |
|------|---------|---------|
| Context Lost 不恢复 | 手动 loseContext + 阻止 restore | 降级 → 壁纸恢复，主程序正常 |
| Shader 编译失败 | 注入无效 GLSL | Fallback 着色器，场景继续渲染 |
| Worker 崩溃 | `worker.terminate()` mid-task | 资源加载标记失败，低模继续运行 |
| NV3D hash 不一致 | 手动篡改 Block | 该 Block 跳过，场景用低模 |
| 内存耗尽 | 循环分配大 ArrayBuffer | OOM 熔断 → 模块卸载 → 主程序正常 |

### 8.3 主程序保护确认

所有崩溃测试之后执行：

- [ ] 软件的游戏库功能正常
- [ ] 设置面板正常
- [ ] 壁纸功能正常
- [ ] 快捷启动正常
- [ ] 数据库查询正常
