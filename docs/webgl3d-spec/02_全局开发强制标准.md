## 一、代码规约

### 1.1 语言与文件组织

| 层 | 语言 | 文件位置 |
|---|------|---------|
| 前端 3D 组件 | TypeScript + JSX | `src/webgl3d/` |
| 前端 3D 状态 | TypeScript (Zustand) | `src/webgl3d/state/` |
| GLSL 着色器 | GLSL ES 3.00（WebGL 2.0） | `src/webgl3d/shaders/` 或独立 `.glsl` 文件 |
| Rust 扩展接口 | Rust | `src-tauri/src/commands/webgl3d.rs` 及相关模块 |
| 打包脚本 | Python 3 | `scripts/webgl3d/` |

### 1.2 命名规范

- 组件文件：PascalCase — `ThreeDCanvas.tsx`, `ShaderLoader.ts`
- 工具函数：camelCase — `createRenderPipeline()`, `loadNv3dManifest()`
- 着色器变量：snake_case — `u_model_matrix`, `v_uv`, `a_position`
- 着色器 Uniform 前缀：`u_`
- 着色器 Attribute 前缀：`a_`
- 着色器 Varying 前缀：`v_`

### 1.3 错误处理

所有跨越模块边界（3D ↔ 主应用）的调用必须显式处理异常：

```typescript
// ✅ 正确
async function loadTheme(themeId: string): Promise<ThemeData | null> {
  try {
    return await bridge.loadNv3dTheme(themeId);
  } catch (e) {
    logError("theme_load_failed", { themeId, error: String(e) });
    return null; // 返回空值，不抛出
  }
}

// ❌ 禁止
async function loadTheme(themeId: string): Promise<ThemeData> {
  return await bridge.loadNv3dTheme(themeId); // 异常直接传播到主应用
}
```

### 1.4 资源清理

任何使用 GPU 资源的对象必须在组件卸载时 dispose：

```typescript
// 组件内
useEffect(() => {
  const renderer = new WebGLRenderer(canvas);
  return () => {
    renderer.dispose(); // 清理所有 GPU 资源
  };
}, []);
```

## 二、依赖管理

### 2.1 依赖原则

- **不新增全局框架**：复用现有 Vite + React + Zustand + Tailwind，不为 3D 模块引入新的构建工具链
- **渲染库选择**：使用轻量级 WebGL 封装库（推荐 Three.js 或其等效替代），不引入游戏引擎
- **按需 import**：着色器编译器、Draco 解码器等重型依赖使用动态 import
- **不引入新的原生依赖**：Tauri Rust 侧不新增系统级依赖

### 2.2 依赖审查

新增任何 npm 包前检查：
1. Bundle 体积增量（`npm run build` 对比）
2. 是否包含原生二进制（`node-gyp`）—— 禁止
3. License 兼容性（AGPL v3 兼容）
4. 是否有更轻的替代方案

## 三、代码分层隔离规范

### 3.1 目录隔离

```
src/webgl3d/                    ← 3D 模块全部代码
├── featureFlag.ts              ← 全局开关（默认 false）
├── index.ts                    ← 唯一公开入口
├── canvas/                     ← 画布组件层
│   ├── ThreeDCanvas.tsx        ← Error Boundary 包裹
│   └── ThreeDErrorBoundary.tsx
├── renderer/                   ← 渲染内核
│   ├── RenderManager.ts        ← 全局单例，唯一 context 持有者
│   ├── SceneManager.ts
│   ├── ResourceCache.ts
│   └── Pipeline.ts
├── shaders/                    ← 着色器
│   ├── common/                 ← 通用（跨主题复用，短期缓存）
│   └── themes/                 ← 主题专属（热切换时释放）
├── interaction/                ← 交互系统
│   ├── RayPicker.ts
│   ├── AnimationController.ts
│   └── EventBus.ts
├── state/                      ← 独立 Zustand store
│   └── threeDStore.ts
├── bridge/                     ← 主应用中转接口
│   ├── useGate.ts              ← 复用主应用权限 hook
│   ├── tauriCommands.ts        ← Tauri invoke 封装
│   └── log.ts                  ← 崩溃日志记录
└── utils/                      ← 模块内部工具
    ├── metrics.ts              ← 性能指标采集（Stats.js 封装）
    └── spector.ts              ← Spector.js 调试接口（仅 dev）
```

### 3.2 禁止跨边界访问

| 方向 | 规则 |
|------|------|
| 3D 模块 → 主应用 | **仅通过 bridge/ 中转**，不直接 import 主应用模块 |
| 主应用 → 3D 模块 | **仅通过 `src/webgl3d/index.ts` 公开的接口**，不深入 import 内部实现 |
| 3D 模块 → Rust | **仅通过 bridge/tauriCommands.ts** 封装调用，不分散 invoke |
| Rust → 3D 模块 | 不直接操作前端状态，仅通过 command 返回值传递数据 |

### 3.3 动态导入

主应用中唯一引用 3D 模块的方式：

```typescript
// 主应用代码中
if (WEBGL3D_ENABLED) {
  const { initThreeD } = await import("@/webgl3d");
  await initThreeD(container);
}
```

## 四、异常熔断开发标准

### 4.1 熔断器状态机

```
CLOSED ──[连续 N 次异常]──→ OPEN ──[冷却 T 秒]──→ HALF_OPEN
                                                      ↓
                                          [尝试恢复，1 次异常即触发]
                                                      ↓
                                       成功 → CLOSED / 失败 → OPEN
```

### 4.2 阈值配置

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 异常阈值 N | 3 | 连续异常次数触发熔断 |
| 冷却时间 T | 60s | OPEN 状态持续时间 |
| 异常窗口 | 30s | 超过此窗口的旧异常不计入连续计数 |
| 致命异常 | — | context_lost（3 次恢复失败）、shader_compile_failed（启动阶段）、OOM — 直接 OPEN 不等待计数 |

### 4.3 熔断触发后行为

1. 卸载 WebGL context 和全部 3D 资源
2. 渲染降级内容（静态 2D 预览图或原生壁纸）
3. 记录崩溃日志（见第六节）
4. 模块状态持久化为 `disabled`
5. 用户可手动重新启用（重置熔断器状态为 CLOSED）

## 五、调试工具链

### 5.1 内置轻量性能面板

开发模式集成 Stats.js，实时展示：

| 指标 | 来源 | 说明 |
|------|------|------|
| FPS | requestAnimationFrame 帧率 | 目标 ≥ 60 |
| Frame Time | 单帧 wall-clock 耗时 | 预算 16.6ms |
| JS Heap | `performance.memory` | 内存趋势监控 |
| DrawCalls | 渲染循环统计 | ≤ 红线值 |
| Triangles | 场景三角面总数 | ≤ 上限值 |
| Textures | GPU 纹理数量 | ≤ 上限值 |

**重要**：性能面板仅在 `import.meta.env.DEV` 下加载，生产构建 tree-shaking 完全移除。

### 5.2 Spector.js 深度调试入口

仅在开发模式可用，手动触发的 WebGL 帧捕获工具：

```typescript
// 开发模式下 console 调用
if (import.meta.env.DEV) {
  (window as any).__spectorCapture = () => {
    // 捕获下一帧的全部 WebGL 调用
  };
}
```

### 5.3 素材自动化校验工具

打包脚本前置校验（见 [10_离线贴图管线](10_离线贴图自动化管线规范.md)）：

| 校验项 | 工具 | 阻断级别 |
|--------|------|---------|
| 模型面数 | glTF-Validator | 超限阻断 |
| 贴图分辨率 | Python/Pillow | 超限警告 |
| 文件命名规范 | Python 正则 | 不规范阻断 |
| 必需文件完整性 | Python 文件检查 | 缺失阻断 |

校验不通过 → 直接阻断 NV3D 打包 → 输出标准化报错日志。

### 5.4 着色器热重载

开发环境支持修改着色器文件后实时刷新：

**依赖**：安装 `vite-plugin-glsl`（将 `.glsl` 文件作为 raw string import，同时注入 HMR 支持）：

```bash
npm install --save-dev vite-plugin-glsl
```

```typescript
// vite.config.ts 新增
import glsl from "vite-plugin-glsl";

export default defineConfig({
  plugins: [
    // ... 现有插件
    glsl(), // 使 .glsl 文件可被 import，并支持 HMR
  ],
});
```

**运行时热重载逻辑**（`ShaderCompiler` 内）：

```
1. Vite HMR 检测到 .glsl 文件变更 → 触发模块热替换
2. ShaderCompiler 收到新的 source string → 重新编译
3. 编译成功 → gl.deleteProgram(old) → 创建新 program → 替换渲染管线中的引用
4. 编译失败 → 控制台报错 → 保留旧 program 继续渲染（不中断运行中的应用）
```

> **注意**：仅 `src/webgl3d/shaders/common/` 下的通用着色器支持 HMR。主题专属着色器（`shaders/themes/`）在主题加载时一次性编译，不参与开发期热重载。

## 六、生产环境崩溃日志规范

### 6.1 日志格式

```
[Nova3D] YYYY-MM-DD HH:MM:SS | LEVEL | CATEGORY | MESSAGE
```

### 6.2 必须记录的事件

| 事件 | 级别 | 内容 |
|------|------|------|
| 模块启动 | INFO | GPU 型号、WebGL 版本、主题 ID |
| 主题加载完成 | INFO | 主题 ID、加载耗时、资源数量 |
| Context Lost | WARN | 触发原因（如睡眠唤醒） |
| Context Restored | INFO | 恢复耗时 |
| Context 恢复失败 | ERROR | 失败原因 |
| Shader 编译失败 | ERROR | Shader 路径、驱动错误信息 |
| 资源加载失败 | ERROR | 资源路径、HTTP 状态码或错误原因 |
| OOM 触发 | ERROR | JS Heap 值、三角面数、纹理内存估算 |
| 熔断触发 | ERROR | 连续异常计数、触发类型 |
| 模块降级 | WARN | 降级目标（如 60→30fps、3D→静态图） |
| 模块卸载 | INFO | 卸载原因（用户手动/自动降级/熔断） |

### 6.3 不应记录的信息

- ❌ 用户文件路径
- ❌ 用户操作日志
- ❌ 网络请求内容
- ❌ 设备唯一标识符
- ❌ 任何可关联到个人身份的 PII

### 6.4 日志存储

- 本地文件，位于应用日志目录（与现有日志合并或独立文件）
- 最多保留 7 天，自动轮转
- 用户可通过设置面板导出日志用于反馈问题

## 七、Feature Flag 规范

### 7.1 定义

```typescript
// src/webgl3d/featureFlag.ts
/** 3D 主题扩展模块全局开关。开发阶段默认 false。 */
export const WEBGL3D_ENABLED = false;
```

### 7.2 使用规则

- 主应用所有对 3D 模块的引用前必须检查此 Flag
- Flag 为 `false` 时不执行 `import("./webgl3d/...")`
- 生产环境 Flag 值来自设置面板的用户选择
- 开发调试时手动设为 `true`，**提交前必须恢复 `false`**

### 7.3 Pre-commit 检查

```
# .husky/pre-commit
grep -q 'WEBGL3D_ENABLED = true' src/webgl3d/featureFlag.ts && \
  echo "ERROR: WEBGL3D_ENABLED must be false before commit" && exit 1
```

### 7.4 验证测试

每次提交前必须通过：

```
1. WEBGL3D_ENABLED = false → npm run tauri:dev → 主应用正常启动
2. 在 3D 模块中故意引入语法错误 → npm run tauri:dev → 主应用仍正常启动
3. npm run typecheck → 全量通过
```

## 八、禁止事项

1. 禁止在 3D 模块中使用 `eval()`、`new Function()`
2. 禁止在着色器中使用无限循环或非确定性分支
3. 禁止在主应用的全局状态中直接引用 3D 模块的类型或实例
4. 禁止绕过 Feature Flag 直接 import 3D 模块
5. 禁止在 Rust 侧为 3D 模块新增系统级原生依赖
6. 禁止在 3D 模块中修改 `window` 全局对象
7. 禁止在 production build 中包含开发调试工具代码

## 九、统一错误码枚举

### 9.1 设计目的

3D 模块所有异常使用统一错误码，确保日志、前端提示、AI 生成的异常处理代码引用同一套语义。错误码分散定义在各文档中容易导致命名冲突和语义不一致。

### 9.2 错误码格式

```
NV3D_{CATEGORY}_{SPECIFIC}
```

- 前缀 `NV3D`：所有 3D 模块错误统一前缀，与主应用错误码区分
- `CATEGORY`：错误大类（4 字母缩写）
- `SPECIFIC`：具体错误，snake_case 大写

### 9.3 错误码枚举

#### 资源加载 (LOAD)

| 错误码 | 含义 | 处理方案 | 来源文档 |
|--------|------|---------|---------|
| `NV3D_LOAD_FILE_OPEN` | NV3D 文件打开失败（不存在/权限不足） | 提示用户检查文件 | 08, 15 |
| `NV3D_LOAD_SIGN_INVALID` | Ed25519 签名无效 | 拒绝加载，提示文件可能损坏 | 08 |
| `NV3D_LOAD_MAGIC_MISMATCH` | Magic bytes 不是 "NV3D" | 拒绝加载，提示格式不支持 | 08 |
| `NV3D_LOAD_MANIFEST_HASH` | Manifest SHA256 与 Footer 不匹配 | 拒绝加载 | 08 |
| `NV3D_LOAD_MANIFEST_JSON` | Manifest JSON 解析失败 | 拒绝加载 | 08, 11 |
| `NV3D_LOAD_BLOCK_HASH` | Resource Block SHA256 不匹配 | 标记该 Block 损坏，使用降级资源 | 08 |
| `NV3D_LOAD_BLOCK_MISSING` | manifest 引用的 Block 在文件中不存在 | 跳过该资源 | 08, 15 |
| `NV3D_LOAD_EXT_UNSUPPORTED` | 主题要求的 WebGL 扩展本机不支持 | 标记主题 unsupported，提示 GPU 不兼容 | 11 |

#### 渲染 (RNDR)

| 错误码 | 含义 | 处理方案 | 来源文档 |
|--------|------|---------|---------|
| `NV3D_RNDR_CTX_CREATE` | WebGL 2.0 context 创建失败 | 标记 unsupported，走静态降级 | 05, 12 |
| `NV3D_RNDR_CTX_LOST` | Context 被动丢失（睡眠/驱动更新） | 进入恢复流程 | 05 |
| `NV3D_RNDR_CTX_RESTORE_FAIL` | Context 恢复失败（5s 超时） | 降级到静态 2D 预览图 | 05 |
| `NV3D_RNDR_SHADER_COMPILE` | Shader 编译失败 | 使用 fallback shader，记录驱动错误 | 06 |
| `NV3D_RNDR_SHADER_TIMEOUT` | Shader 编译超过 3s | 中止编译，使用 fallback | 06, 08 |
| `NV3D_RNDR_SHADER_LINK` | Shader program link 失败 | 使用 fallback program | 06 |
| `NV3D_RNDR_FRAME_TIMEOUT` | 连续帧时间超预算 | 触发降级评估（降帧/降画质） | 05, 19 |

#### 内存/资源 (MEM)

| 错误码 | 含义 | 处理方案 | 来源文档 |
|--------|------|---------|---------|
| `NV3D_MEM_OOM_WARNING` | JS Heap 快速增长 | 主动释放闲置资源，触发 GC | 08 |
| `NV3D_MEM_OOM_CRITICAL` | OOM 熔断触发 | 卸载模块，恢复壁纸，通知用户 | 02, 08 |
| `NV3D_MEM_LEAK_DETECTED` | 资源切换后计数异常增长 | 自动全量 dispose + gl.flush() | 19 |
| `NV3D_MEM_CACHE_OVERFLOW` | 缓存目录超过用户配置上限 | LRU 淘汰非激活主题缓存 | 08 |

#### Worker (WKR)

| 错误码 | 含义 | 处理方案 | 来源文档 |
|--------|------|---------|---------|
| `NV3D_WKR_CRASH` | Worker 线程崩溃 | 标记该批次资源加载失败，使用低精度替代 | 08 |
| `NV3D_WKR_DECODE_FAIL` | Draco/KTX2 解码失败 | 标记该 Block 失败，使用低精度替代 | 08 |
| `NV3D_WKR_INIT_FAIL` | Worker 初始化失败 | 降级到单 Worker 模式或主线程解码 | 08 |

#### 接口/权限 (AUTH)

| 错误码 | 含义 | 处理方案 | 来源文档 |
|--------|------|---------|---------|
| `NV3D_AUTH_DISABLED` | Feature Flag 为 false | 不加载 3D 模块 | 02 |
| `NV3D_AUTH_PREMIUM_REQUIRED` | 当前用户非 Member | UI 显示升级引导 | 01, 15 |
| `NV3D_AUTH_LICENSE_EXPIRED` | Member 已过期 | 禁用模块，保留数据和下载文件 | 01 |

#### 素材/校验 (ASST)

| 错误码 | 含义 | 处理方案 | 来源文档 |
|--------|------|---------|---------|
| `NV3D_ASST_FACE_COUNT` | 模型三角面数超过预算 | 阻断打包 | 09, 17 |
| `NV3D_ASST_RESOLUTION` | 贴图分辨率超上限 | 阻断打包 | 09, 17 |
| `NV3D_ASST_NAMING` | 文件命名不符合规范 | 阻断打包 | 09, 17 |
| `NV3D_ASST_MISSING_FILE` | 必需文件缺失 | 阻断打包 | 17 |
| `NV3D_ASST_REF_INVALID` | manifest 中资源引用指向不存在的 key | 阻断打包 | 11 |

#### 交互 (INTR)

| 错误码 | 含义 | 处理方案 | 来源文档 |
|--------|------|---------|---------|
| `NV3D_INTR_CONFIG_PARSE` | 交互配置解析失败 | 该交互跳过，其他交互正常 | 07 |
| `NV3D_INTR_ANIM_MISSING` | 动画文件缺失 | 跳过动画，执行后续动作 | 07 |
| `NV3D_INTR_CONDITION_ERROR` | 条件评估报错 | 视为条件不满足，不触发 | 07 |
| `NV3D_INTR_ACTION_FAILED` | 动作执行中途异常 | 跳过该动作，执行下一个 | 07 |

### 9.4 使用规则

- 日志记录必须包含错误码，方便搜索和统计：
  ```
  [Nova3D] 2026-07-22 14:30:00 | ERROR | NV3D_RNDR_SHADER_COMPILE | shaders/postprocess.glsl: ERROR: 0:15: 'u_unknown' : undeclared identifier
  ```
- 前端给用户的提示文案通过错误码映射到 i18n key，不直接在代码中硬编码中文/英文：
  ```typescript
  const userMessage = i18n.t(`error.${errorCode}`); // error.NV3D_LOAD_SIGN_INVALID
  ```
- 新增加错误码时，先在本文档注册，再在对应模块中使用。避免各模块自行发明错误码。
- 错误码不覆盖主应用已有的错误体系——仅用于 `src/webgl3d/` 范围内的异常。
