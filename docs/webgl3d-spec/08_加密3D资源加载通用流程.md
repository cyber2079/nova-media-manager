## 一、NV3D 容器格式

### 1.1 格式总览

.nv3d 文件为 WebGL 交互主题专属容器，与普通主题 .nvtp 格式完全独立。

```
┌─ 0 ────────────────────────────────────────┐
│  Magic: "NV3D" (4B)                        │  ← 全新魔数，无歧义
├─ 4 ────────────────────────────────────────┤
│  Header (64B)                              │
│  - version: u16 LE = 1                     │
│  - flags: u16 LE                           │
│    · bit 0: encrypted (AES-256-GCM)        │
│    · bit 1: signed (Ed25519)               │
│    · bit 2-15: reserved                    │
│  - manifestSize: u32 LE                    │
│  - manifestHash: [u8; 32] (SHA256)         │
│  - blockCount: u16 LE                      │
│  - reserved: [u8; 22]                      │
├─ 68 ───────────────────────────────────────┤
│  Manifest Index (JSON, gzip 压缩)           │
│  长度为 manifestSize bytes                 │
├─ 68 + manifestSize ────────────────────────┤
│  Resource Block 1                          │
│  - blockType: u8 (0=raw, 1=gzip, 2=store)  │
│  - blockSize: u32 LE                       │
│  - blockHash: [u8; 32] (SHA256)            │
│  - data: [u8; blockSize]                   │
├─ ... ──────────────────────────────────────┤
│  Resource Block N                          │
├─ EOF - 136 ─────────────────────────────────┤
│  Footer (136B)                             │
│  - manifestSize: u32 LE (4B, 冗余)         │
│  - manifestHash: [u8; 32] (SHA256)         │
│  - contentHash: [u8; 32] (签名覆盖范围)     │
│  - signature: [u8; 64] (Ed25519)           │
│  - magic: "NV3D" (4B, 尾部标记)            │
└────────────────────────────────────────────┘
```

### 1.2 关键设计决策

| 决策 | 说明 |
|------|------|
| Manifest 紧随 Header | 读完头部 68B 即可获取全部资源索引，无需尾部定位 |
| 每 Block 独立压缩 | 随机读取单 Block 无需解压整个包 |
| 已压缩格式用 `store` | Draco/KTX2 已高效压缩，不重复压缩浪费 CPU |
| 每 Block 独立 SHA256 | 损坏定位到具体资源，非整个包作废 |
| 尾部冗余 | Footer 包含 Manifest 的 hash + size，支持尾部定位 |
| AES-256-GCM 加密 | flags bit 0 控制，密钥派生复用现有体系 |
| Ed25519 签名 | flags bit 1 控制，64B 签名位于 Footer，复用现有公钥 |

### 1.3 与普通主题 .nvtp 的区分

| 格式 | 扩展名 | 魔数 | 用途 |
|------|--------|------|------|
| NVTP (普通主题) | .nvtp | NVTP | CSS + 图片 + 音频桌面主题 |
| NV3D (3D 主题) | .nv3d | NV3D | glTF 模型 + GLSL 着色器 + 交互场景 |

客户端根据扩展名 + 魔数双重识别，互不干扰。旧版客户端收到 .nv3d 文件直接提示"不支持此文件类型"。

## 二、分层加载策略

### 2.1 加载优先级

```
Phase 1 — 即刻展示（< 500ms）
  ├─ 读取 Header + Manifest
  ├─ 解析主题元数据
  ├─ 展示主题名称 + 封面图（从 previews 区块）
  └─ 用户可以感知"正在加载什么"

Phase 2 — 首帧可交互（< 5s）
  ├─ 加载低模场景（scene_low）
  ├─ 加载低分辨率贴图（如 1024×1024）
  ├─ 构建基础场景 + 灯光 + 相机
  └─ 用户看到可旋转的 3D 场景

Phase 3 — 后台渐进增强（不限时，非阻塞）
  ├─ Worker 异步加载高精度模型（scene_hd）
  ├─ Worker 异步加载高分辨率贴图
  ├─ 逐步替换场景中的低精度资源
  └─ 最终达到完整画质
```

### 2.2 Phase 间通信

```
Main Thread                          Web Worker
    │                                    │
    ├── postMessage({load, block_id})───→│
    │                                    ├── 解压资源
    │                                    ├── 解码纹理
    │                                    ├── 解码 Draco
    │                                    │
    │←── postMessage({ready, block_id})──┤
    │                                    │
    ├── transfer ArrayBuffer ←──────────── (Transferable, 零拷贝)
    │
    ├── 上传到 GPU → 替换低精度资源
```

## 三、双缓存策略

### 3.1 临时目录缓存

```
{app_data_dir}/
└── webgl3d_cache/
    ├── cyber-girl-apartment/
    │   ├── manifest.json        ← 已解析的 manifest
    │   ├── models/              ← 解压后的模型文件
    │   ├── textures/            ← 解压后的贴图文件
    │   └── .meta                ← 缓存元信息（最后访问时间、大小）
    └── ink-wash/
        └── ...
```

**规则**：
- .nv3d 首次加载时，按需解压 Block 到临时目录
- 后续加载同一主题 → 先检查缓存 → 命中则直接读文件
- 缓存命中时仍需校验文件 hash（防磁盘损坏）

### 3.2 内存缓存

仅当前激活主题的可见资源驻留在 GPU/JS 内存：
- 当前场景模型 + 贴图 → GPU 显存
- 下一场景预加载候选 → JS ArrayBuffer（< 50MB 限额）
- 闲置超过 30s 的高精度资源 → 从内存释放，回退到磁盘缓存

### 3.3 缓存容量管理

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 缓存总容量上限 | 2 GB | 用户可配置（1-10 GB） |
| 单主题缓存上限 | 1 GB | 超出则淘汰最旧的非激活主题 |
| 淘汰策略 | LRU | 最后访问时间排序 |
| 当前激活主题 | 不淘汰 | 除非正在切换 |

## 四、Web Worker 后台处理

### 4.1 Worker 职责

```
Web Worker Pool（最多 2 个 Worker 实例）
    │
    ├─ Worker 1：资源解压
    │   ├─ gzip decompress
    │   └─ 提取原始 bytes
    │
    └─ Worker 2：格式解码
        ├─ Draco 网格解码
        ├─ KTX2/Basis 纹理转码 → RGBA8
        └─ 返回 Transferable ArrayBuffer
```

### 4.2 Worker 生命周期

- 主题加载开始时创建 Worker 实例
- 主题切换时终止旧 Worker（`worker.terminate()`），创建新的
- Worker 崩溃 → 主线程收到 error 事件 → 标记该资源加载失败 → 使用低精度替代
- 模块卸载时终止全部 Worker

### 4.3 主线程保护

Worker 中的任何异常不会传播到主线程：
```typescript
worker.onerror = (event) => {
  logError("worker_crash", { message: event.message });
  // 不重新抛出，主线程继续运行
};
```

### 4.4 Worker 间数据传递协议

两个 Worker 之间为**流水线**关系（解压 → 解码），数据通过**主线程中转 + Transferable 零拷贝**传递：

```
Worker 1 (解压)                Main Thread               Worker 2 (解码)
    │                              │                          │
    ├─ 解压 Block → ArrayBuffer    │                          │
    ├─ postMessage({               │                          │
    │    type: "decompressed",     │                          │
    │    blockId,                  │                          │
    │    buffer: ArrayBuffer       │                          │
    │  }, [buffer])  ─────────────→│                          │
    │                              ├─ 根据 blockId 查         │
    │                              │  manifest 中声明的格式   │
    │                              │ (draco/ktx2/raw)         │
    │                              ├─ postMessage({           │
    │                              │    type: "decode",       │
    │                              │    format,               │
    │                              │    buffer                 │
    │                              │  }, [buffer]) ──────────→│
    │                              │                          ├─ Draco 解码
    │                              │                          │  或 KTX2 转码
    │                              │                          ├─ postMessage({
    │                              │                          │    type: "decoded",
    │                              │                          │    blockId,
    │                              │                          │    buffer (RGBA8)
    │                              │                          │  }, [buffer]) ──→
    │                              │                          │
    │                              ├─ 上传到 GPU              │
    │                              ├─ 替换低精度资源          │
```

**关键规则**：
- 所有 ArrayBuffer 通过 `postMessage` 的 transfer list 传递，**零拷贝**
- 主线程不持有解压后数据的副本——收到即转发给 Worker 2，或直接上传 GPU
- 对于 `store` 类型的 Block（如 KTX2、Draco GLB），Worker 1 不解压，直接原样传递给 Worker 2
- 若 Worker 2 崩溃，主线程将当前批次中未处理的 Block 标记为失败，使用低精度替代——不重试整个批次
- Worker 1 不关心 Worker 2 的存活状态，解压完成后只管发送；主线程负责路由和容错

### 4.5 单 Worker 降级模式

当资源量较小或 Worker 2 初始化失败时，可降级为单 Worker 模式：Worker 1 解压 → 主线程直接读取原始 bytes（不做 Draco/KTX2 转码）→ 使用浏览器内置解码器（如 `ImageBitmap` for 贴图、`DRACOLoader` on main thread for 模型）。此时牺牲部分主线程时间换取健壮性。

```
.nv3d 文件打开
    │
    ▼
[1] Ed25519 签名验证（Rust 侧，Footer 中 64B signature）
    ├─ 通过 → 继续
    └─ 失败 → 拒绝加载
    │
    ▼
[2] Magic Bytes 验证
    ├─ "NV3D" → NV3D 解析器
    └─ 其他   → 拒绝加载
    │
    ▼
[3] Manifest 完整性校验
    └─ Footer 中的 manifestHash 与实际 SHA256 比对
    │
    ▼
[4] Manifest JSON Schema 校验
    └─ 按 11_主题Schema 规范严格校验
    │
    ▼
[5] 每个 Resource Block 加载时校验 SHA256
    └─ 不匹配 → 该 Block 标记为损坏 → 使用降级资源
    │
    ▼
[6] Shader 编译超时检测
    └─ 超过 3s → 中止 → 使用 fallback shader
```

### 5.2 签名体系

NV3D 使用 Ed25519 签名，复用现有公钥体系：
- 主题包使用私钥签名，64B 签名嵌入 Footer
- 客户端使用内置公钥验证
- 签名无效 → 拒绝加载 → 用户看到"文件无效或不完整"

### 5.3 元数据安全

- JSON 解析使用 `JSON.parse()`，不使用 `eval()`
- 文本内容不通过 `dangerouslySetInnerHTML` 注入 DOM
- 所有用户可见文本通过安全的 React 文本节点渲染

## 六、OOM 熔断机制

### 6.1 检测信号

| 信号 | 阈值 | 来源 |
|------|------|------|
| JS Heap 快速增长 | 30s 内增长 > 200MB | `performance.memory` |
| 分配失败 | `try { new ArrayBuffer(N) } catch` | 主动探测 |
| Canvas context lost | `webglcontextlost` 事件 (OOM 原因) | 被动监听 |

### 6.2 熔断响应

```
OOM 信号触发
    ↓
1. 立即暂停渲染循环
2. 释放全部非必要内存缓存
3. 降至最低画质级别
4. 尝试恢复渲染
    ├─ 恢复成功 → 维持低画质，记录 OOM 事件
    └─ 恢复失败（30s 内仍 OOM）
         ↓
       卸载全部 3D 模块
         ↓
       恢复原生壁纸
         ↓
       通知用户 + 记录崩溃日志
```

### 6.3 主动内存管理

除被动熔断外，主动管理策略：
- 每 30s 检查 JS Heap 趋势
- 上升斜率异常 → 主动触发 GC（将大型数组置 null）
- 闲置资源超过 30s 未使用 → 自动释放
