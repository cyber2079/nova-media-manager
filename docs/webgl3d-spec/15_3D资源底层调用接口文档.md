# 15 — 3D 资源底层调用接口文档

> **优先级**：P1
> **版本**：v1.0 | **更新日期**：2026-07-22
> **覆盖**：前端调用 Rust 侧接口的入参/返回值/异常处理、NV3D 文件导入/解析/卸载接口、中转封装规则
> **不覆盖**：Rust 侧具体实现细节、Tauri command 注册方式（依托现有工程自动适配）

---

## 一、接口设计原则

1. **中转封装**：所有前端调用不直接触及 Rust 核心函数，通过 `bridge/tauriCommands.ts` 统一封装
2. **失败返回空值**：所有接口调用失败时返回 `null` 或默认安全值，**不抛出异常到调用方**
3. **无状态设计**：每个接口调用自包含上下文，不依赖 Rust 侧 session 状态
4. **异步非阻塞**：所有文件 I/O 操作走 Tauri async command，不阻塞主线程

## 三、接口详细定义

### 3.1 nv3d_open — 打开 NV3D 文件

```
前端调用
  invoke("nv3d_open", { path: string })
       │
       ▼
Rust 侧
  1. 打开文件
  2. 读取 Magic Bytes → 确定 v1/v2
  3. 验证签名
  4. 解析 Header + Manifest
  5. 返回 Manifest JSON 字符串
```

**入参**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `path` | string | NV3D 文件绝对路径 |

**返回值**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `success` | boolean | 是否成功 |
| `manifest` | string? | Manifest JSON 字符串（成功时） |
| `formatVersion` | string? | `"1.0"` 或 `"2.0"` |
| `error` | string? | 错误描述（失败时） |

**前端封装**：

```typescript
// bridge/tauriCommands.ts
async function openNv3d(path: string): Promise<Manifest | null> {
  try {
    const result = await invoke<{ success: boolean; manifest?: string; error?: string }>(
      "nv3d_open", { path }
    );
    if (!result.success || !result.manifest) {
      logError("nv3d_open", result.error ?? "unknown");
      return null;
    }
    return JSON.parse(result.manifest) as Manifest;
  } catch (e) {
    logError("nv3d_open", String(e));
    return null;
  }
}
```

### 3.2 nv3d_read_block — 读取 Resource Block

```
前端调用
  invoke("nv3d_read_block", { path, blockId, expectedHash })
```

**入参**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `path` | string | NV3D 文件绝对路径 |
| `blockId` | string | 资源在 manifest 中的 key（如 `"models/scene_low"`） |
| `expectedHash` | string | manifest 中声明的 SHA256 |

**返回值**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `success` | boolean | 是否成功 |
| `data` | number[]? | 原始字节数组（成功时） |
| `hashMatch` | boolean? | hash 是否匹配 |
| `error` | string? | 错误描述（失败时） |

> **注意**：实际传输使用 Tauri 二进制通道（ArrayBuffer），此处 `number[]` 仅为逻辑类型描述。大块二进制数据不得通过 JSON 序列化传输。

**前端封装**：

```typescript
async function readBlock(
  path: string, blockId: string, expectedHash: string
): Promise<ArrayBuffer | null> {
  try {
    const result = await invoke<{
      success: boolean; data?: number[]; hashMatch?: boolean; error?: string;
    }>("nv3d_read_block", { path, blockId, expectedHash });

    if (!result.success || !result.data) {
      logError("nv3d_read_block", `${blockId}: ${result.error ?? "unknown"}`);
      return null;
    }
    if (!result.hashMatch) {
      logError("nv3d_read_block", `${blockId}: hash mismatch`);
      return null;
    }
    return new Uint8Array(result.data).buffer;
  } catch (e) {
    logError("nv3d_read_block", String(e));
    return null;
  }
}
```

### 3.3 nv3d_verify — 校验签名

在打开之前验证 NV3D 文件签名有效性：

**入参**：`{ path: string }`

**返回值**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `valid` | boolean | 签名是否有效 |
| `formatVersion` | string? | 格式版本 |
| `error` | string? | 无效原因 |

### 3.4 webgl3d_save_data — 保存用户存档

```
前端调用
  invoke("webgl3d_save_data", { themeId, slot, data })
```

**入参**：

| 参数 | 类型 | 说明 |
|------|------|------|
| `themeId` | string | 主题 ID |
| `slot` | number | 存档槽位（0-9） |
| `data` | string | 存档 JSON 字符串 |

**返回值**：`{ success: boolean; error?: string }`

**规则**：
- 写入独立数据表 `webgl3d_user_data`
- 每个 themeId + slot 唯一，重复写入覆盖
- 不修改原有数据库表

---

### 3.5 webgl3d_load_data — 读取用户存档

**入参**：`{ themeId: string; slot: number }`

**返回值**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `success` | boolean | 是否成功 |
| `data` | string? | 存档 JSON 字符串 |
| `exists` | boolean | 存档是否存在 |

- 存档不存在 → `exists: false`，前端使用默认初始状态
- 读取失败 → `success: false`，前端记录日志，使用默认状态

---

### 3.6 webgl3d_delete_data — 删除存档

**入参**：`{ themeId: string; slot?: number }`

- 不传 `slot`：删除该主题全部存档
- 传 `slot`：仅删除指定槽位

**返回值**：`{ success: boolean; deletedCount: number; error?: string }`

---

### 3.7 webgl3d_cache_size — 查询缓存大小

**入参**：无（或 `{ themeId?: string }` — 可选，查询特定主题缓存）

**返回值**：`{ success: boolean; sizeBytes: number; themeCount: number }`

- 供设置面板展示缓存占用
- 供 LRU 淘汰逻辑使用

---

### 3.8 webgl3d_clear_cache — 清理缓存

**入参**：`{ themeId?: string }`

- 不传 `themeId`：清理全部 3D 缓存
- 传 `themeId`：仅清理指定主题缓存
- 当前激活主题被清理时 → 仅清理非活跃资源，保留下次启动可快速恢复的 manifest 缓存

**返回值**：`{ success: boolean; freedBytes: number; error?: string }`

## 四、中转封装规则

### 4.1 封装层职责

`bridge/tauriCommands.ts` 是前端与 Rust 之间的唯一通信通道：

```typescript
// ✅ 正确：通过封装层调用
import { openNv3d, readBlock, saveData } from "@/webgl3d/bridge/tauriCommands";

// ❌ 禁止：在封装层之外直接 invoke
import { invoke } from "@tauri-apps/api/core";
invoke("nv3d_open", { path }); // 禁止分散调用
```

### 4.2 统一错误处理

```typescript
// 所有封装函数必须遵循此模式
async function wrappedCommand<T>(command: string, args: Record<string, unknown>): Promise<T | null> {
  try {
    const result = await invoke<T>(command, args);
    return result;
  } catch (e) {
    logError(command, String(e));
    return null; // 永远返回 null，不抛出
  }
}
```

### 4.3 权限检查前置

涉及文件读写的接口在调用前检查用户权限：

```typescript
async function openNv3d(path: string): Promise<Manifest | null> {
  if (!useSettingsStore.getState().webgl3dEnabled) return null;
  if (!useGate("premium-theme")) return null;
  // ... 实际调用
}
```

## 五、Rust 侧约定

### 5.1 新增文件

所有 3D 相关 Tauri command 放在独立的模块文件中：

```
src-tauri/src/commands/webgl3d.rs
```

或作为模块目录：
```
src-tauri/src/commands/webgl3d/
├── mod.rs
├── nv3d.rs     # NV3D 解析
├── cache.rs    # 缓存管理
└── data.rs     # 存档读写
```

### 5.2 错误处理

每个 command 返回 `Result<OkData, String>`，Tauri 框架自动转换为前端可接收的格式：

```rust
#[tauri::command]
fn nv3d_open(path: String) -> Result<Nv3dOpenResponse, String> {
    // 每个 command 内部必须用 `std::panic::catch_unwind` 包裹可能 panic 的代码路径，
    // 或确保所有代码路径不出现 panic。Tauri 默认不会捕获 Rust panic。
    // Result::Err 被序列化为 { success: false, error: "..." }
    // 不影响其他 command 的功能
}
```

### 5.3 依赖约束

- 不新增系统级原生依赖
- NV3D 解析使用纯 Rust 实现（不依赖 C 库）
- 加密部分复用现有 `license/` 和 `theme/crypto.rs` 模块：密钥派生复用，加密算法使用 AES-256-GCM
